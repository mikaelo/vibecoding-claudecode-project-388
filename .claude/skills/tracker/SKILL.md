# tracker

Собирает snapshot текущих цен по всем бумагам одного счёта Финам. Для получения
цены по каждой бумаге **вызывает skill `extract_price`** — не дублирует его логику
и не обращается к `get-quote` напрямую.

## Настройки

Значения по умолчанию и **правило записи snapshot** бери из `KNOWLEDGE.md` в
корне репозитория (счёт по умолчанию, репозиторий-хранилище и ветка, шаблон
имени файла, пороги `snapshot_deviation_threshold`=10% и
`snapshot_open_change_threshold`=1%). Явно переданные аргументы имеют приоритет.

Snapshot содержит **все** бумаги счёта и **всегда записывается полностью** в
репозиторий-хранилище. Условия из раздела «Правило записи и возврата snapshot»
в `KNOWLEDGE.md` определяют лишь то, **какие бумаги skill возвращает** в ответе
(см. шаги 5–7).

## Вход

- `account` — идентификатор счёта одним из двух способов:
  - **ID счёта** (напр. `869168`), либо
  - **название счёта** (`label`) или торговый код (`tradeCode`).
  - Если `account` не передан — используй счёт по умолчанию из `KNOWLEDGE.md`.

## Выход

**Важно:** в репозиторий-хранилище пишется **полный** snapshot (все бумаги), а в
ответе skill возвращает **только** бумаги, попавшие под условия (см. `KNOWLEDGE.md`,
раздел «Правило записи и возврата snapshot»).

1. **JSON-результат:**

```json
{
  "account_id": "869168",
  "account_label": "",
  "generated_at": "2026-07-14T12:30:00Z",
  "saved_file": "2026-07-14-12-30.json",
  "saved_rows_count": 8,
  "rows": [
    { "ticker": "BAZA@MISX", "open": 94.8, "current": 84.5, "datetime": "2026-07-14T15:28:58+03:00" }
  ]
}
```

- `rows` — **только бумаги под условиями** (отклонение от цены покупки > 10% И
  изменение цены открытия относительно предыдущего snapshot > 1%). Поля берутся
  из ответа `extract_price`; `datetime` — по московскому времени (МСК, UTC+3),
  ISO 8601 со смещением `+03:00`.
- `saved_file` — имя записанного файла полного snapshot; `saved_rows_count` —
  сколько бумаг в сохранённом (полном) snapshot.
- Бумаги, по которым `extract_price` вернул ошибку, в фильтр возврата не
  попадают (нет `open`/`current` для сравнения), но в полном сохранённом snapshot
  сохраняются как `{ "ticker": "...", "error": "<текст>" }`.

2. **Markdown-таблица** возвращаемых бумаг (для читаемости), например:

| Тикер     | Открытие | Текущая | Время (МСК)              |
|-----------|---------:|--------:|--------------------------|
| BAZA@MISX |     94.8 |    84.5 | 2026-07-14T15:28:58+03:00 |

## Шаги

1. **Резолв счёта.** Вызови Finam MCP `get-accounts-list`
   (`mcp__Finam_MCP__get-accounts-list`). Сопоставь вход `account`:
   - точное совпадение с `id` элемента → берём этот `id`;
   - иначе точное (регистронезависимо) совпадение с непустым `label` → его `id`;
   - иначе совпадение с `tradeCode` → его `id`;
   - если ничего не совпало — верни `{ "error": "Счёт не найден", "account": <вход> }`.
2. **Позиции.** Вызови `get-account` (`mcp__Finam_MCP__get-account`) с найденным
   `account_id`.
   - Если ответ содержит `error` (напр. «Unknown account type») — верни
     `{ "account_id": ..., "error": "<сообщение>" }`.
   - Если `positions` отсутствует/пуст — верни snapshot с пустым `rows`.
   - Иначе по каждой позиции возьми `symbol` (формат уже `TICKER@MIC`) и
     **среднюю цену покупки** `averagePrice` — это вложенный объект,
     `averagePrice.value` (строка, напр. `"113.29"`), приведи к числу. Она нужна
     для условия записи (шаг 5).
3. **Цены через skill `extract_price`.** Для КАЖДОГО `symbol`:
   - вызови skill `extract_price` через инструмент **Skill**
     (`skill: extract_price`, `args: <symbol>`);
   - возьми из его результата `{ ticker, open, current, datetime }` и добавь в
     `rows`. При ошибке — добавь `{ ticker, error }` и продолжай.
   Не вызывай `get-quote` и не повторяй нормализацию/парсинг из `extract_price` —
   вся логика получения цены живёт в `extract_price`.
4. **Собери snapshot.** Заполни `account_id`, `account_label` (из шага 1),
   `generated_at` (текущее время UTC ISO 8601) и `rows` — **по всем бумагам**
   счёта (строки не фильтруются).
5. **Найди предыдущий snapshot** (до записи текущего). Через GitHub MCP
   `get_file_contents` перечисли корень репозитория-хранилища
   (`owner`/`repo`/`branch` из `KNOWLEDGE.md`) и выбери самый свежий файл
   `YYYY-MM-DD-HH-MM.json` (имена сортируются хронологически). Прочитай его и
   построй по тикерам карту `{ open, current }`, а также запомни его
   `generated_at`. Определи **день** предыдущего и текущего snapshot — календарную
   дату **по МСК** (из `generated_at` в UTC + 3 часа). Если репозиторий пуст —
   карта пустая (первый запуск).
6. **Запиши полный snapshot в репозиторий — всегда.** Сохрани JSON-snapshot из
   шага 4 (со **всеми** бумагами) в репозиторий-хранилище (`repo`/`branch` из
   `KNOWLEDGE.md` — по умолчанию `mikaelo/tracker-data`, ветка `main`):
   - имя файла — по шаблону из `KNOWLEDGE.md` (`YYYY-MM-DD-HH-MM.json`) в
     **корне** репозитория, где дата/время берутся из `generated_at` в **UTC**
     (напр. `2026-07-14-12-41.json`);
   - содержимое — JSON-snapshot из шага 4 (тот же объект);
   - запись **напрямую в `main`**, без создания веток и без PR — snapshot'ы
     складываются прямо в основную ветку репозитория;
   - запись через GitHub MCP `create_or_update_file` с параметрами
     `owner`/`repo`/`branch` из `KNOWLEDGE.md` (по умолчанию `owner: mikaelo`,
     `repo: tracker-data`, `branch: main`), `path: <YYYY-MM-DD-HH-MM>.json`,
     `content: <JSON snapshot>`, `message: "snapshot <account_id> <generated_at>"`.
     Имя уникально в пределах минуты → это всегда создание нового файла, `sha`
     не нужен.
   - Предусловие: репозиторий-хранилище должен быть в scope сессии
     (при необходимости подключить его через `add_repo`) и доступен на запись.
     Если запись вернула ошибку доступа (403/недоступен) — сообщи об этом, но
     результат всё равно верни (см. ниже).
7. **Отфильтруй бумаги для возврата** по условиям из `KNOWLEDGE.md` («Правило
   записи и возврата snapshot»). Для КАЖДОЙ бумаги полного snapshot (с числовыми
   `open`/`current`, без `error`) проверь **ОБА** условия:
   1. `abs(averagePrice − current) / averagePrice > snapshot_deviation_threshold`
      (10%; `averagePrice` — из шага 2);
   2. изменение относительно предыдущего snapshot больше
      `snapshot_change_threshold` (1%), **базис зависит от дня** (шаг 5):
      - **разные дни** → по цене открытия:
        `abs(open − prev_open) / prev_open > snapshot_change_threshold`;
      - **один день** → по текущей цене:
        `abs(current − prev_current) / prev_current > snapshot_change_threshold`.
      Если предыдущего значения для тикера нет (первый запуск / новая бумага) —
      условие 2 считается выполненным.
   В возврат попадают только бумаги, прошедшие **оба** условия.
8. **Отдай результат.** Верни JSON-результат (со `saved_file`, `saved_rows_count`
   и отфильтрованными `rows`) и Markdown-таблицу возвращаемых бумаг, а также
   статус записи в репозиторий (путь к созданному файлу либо текст ошибки). Если
   под условия не попала ни одна бумага — верни пустой `rows` (полный snapshot
   при этом всё равно сохранён).

## Замечания

- В репозиторий пишется **полный** snapshot (все бумаги: тикер — цена открытия —
  текущая — время). Средняя цена покупки (`averagePrice`) в файл не входит, но
  используется для фильтра возврата (условие 10%).
- Разделение «сохраняем всё / возвращаем отфильтрованное» намеренное: полный
  snapshot нужен как база для сравнения по цене открытия при следующем запуске, а
  в ответе показываются только значимо изменившиеся бумаги за порогом 10%.
- Один вызов = один счёт. Для нескольких счетов вызывай `tracker` по каждому.
